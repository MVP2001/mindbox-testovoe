Задача
Спроектировать размещение веб-приложения в Kubernetes-кластере со следующими условиями:
Инфраструктура: 3 зоны, 5 нод
Приложение: требует около 5-10 секунд для инициализации
Нагрузка: дневной цикл (пик днём — нужно 4 пода, ночью — минимум)
Ресурсы: на первые запросы приложению требуется значительно больше ресурсов CPU, в дальнейшем потребление ровное в районе 0.1 CPU. По памяти всегда “ровно” в районе 128M memory
Цель: максимальная отказоустойчивость при минимальных затратах ресурсов
Ключевые вызовы
1. Распределение по зонам (Anti-Affinity)
Проблема: При 4 подах и 3 зонах математически невозможно равномерное распределение (остаётся "лишний" под). Жёсткое требование "по одному поду на зону" приводит к тому, что 4-й под не может запуститься.
Решение: Использовал мягкое распределение (topologySpreadConstraints с maxSkew: 1). Это позволяет схеме 2-1-1 (два пода в одной зоне, по одному в других) и гарантирует, что при падении любой зоны приложение продолжает работать.
2. Ночная экономия vs Отказоустойчивость
Проблема: Классический подход "ночью оставляем 1 под" экономит ресурсы, но создаёт конфликт: если ночью упадёт зона с этим единственным подом — сервис ляжет. Кроме того, система безопасности Kubernetes (PodDisruptionBudget) блокирует обновления нод, требуя минимум 2 работающих пода.
Решение: Установил ночной минимум в 2 пода. Это:
На 60% дешевле, чем держать 4 пода круглосуточно
Позволяет пережить падение зоны ночью (останется 1 под, затем автоматически восстановится до 2)
Не блокирует техническое обслуживание кластера (обновления нод)
3. Стартовый всплеск ресурсов
Проблема: При старте приложению нужно много CPU (инициализация), но затем потребление падает до 0.1 CPU. Если дать много ресурсов "про запас" — переплата. Если дать мало — приложение будет "душиться" при старте.
Решение: Гибкая схема запросов ресурсов:
Запрос (Request): 0.1 CPU — столько приложение гарантированно получает и столько реально потребляет в работе
Лимит (Limit): 0.5 CPU — позволяет "выпрыгнуть" выше запроса на старте, но не съест всю ноду
Память зафиксировал строго по ТЗ (128Мб) — она стабильна, без всплесков.
4. Проверка здоровья приложения
Проблема: Стандартные проверки (liveness/readiness) начинают работать сразу, но приложение 5-10 секунд не отвечает из-за инициализации. Kubernetes может принять это за "зависание" и перезапустить под, создавая бесконечный цикл перезагрузок.
Решение: Трёхуровневая система проверок:
StartupProbe — даёт 30 секунд на инициализацию, не мешая старту
ReadinessProbe — после успешного старта проверяет готовность принимать трафик
LivenessProbe — проверяет, не зависло ли приложение в процессе работы
5. Плавное обновление без downtime
При rolling update новый под стартует рядом со старыми, затем старый получает команду на выключение. Настроил "мягкое" выключение: 15 секунд на завершение текущих запросов перед остановкой контейнера.
Итоговая архитектура
Днём (пик):
4 пода, распределены по зонам 2-1-1
Потребление: ~0.4 CPU суммарно (4 × 0.1)
Выдерживает падение любой зоны (останется 3 пода в 2 зонах)
Ночью (минимум):
2 пода в разных зонах
Потребление: 0.2 CPU суммарно
Экономия 50% по сравнению с круглосуточной работой на 4 подах
Переживает падение зоны (автовосстановление до 2 подов)
Автомасштабирование:
Утром: быстрое наращивание до 4 подов при росте нагрузки (за 1-2 минуты)
Вечером: плавное снижение (чтобы не "дёргаться" при временных просадках трафика)
